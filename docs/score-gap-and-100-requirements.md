## 1. 목적

본 문서는 기존 임원 보고서의 5개 평가항목 점수가 왜 현재 수준에 머물렀는지 세부적으로 해부하고, 각 항목을 **100점 수준**으로 끌어올리기 위해 필요한 요구사항을 기능·비기능·운영·보안 관점에서 명세한다.

이 문서의 목적은 다음 두 가지다.

1. 현재 감점 사유를 모호하지 않게 수치와 항목으로 설명한다.
2. 제품화 팀이 바로 백로그와 로드맵으로 옮길 수 있도록 요구사항을 구조화한다.

## 3. 항목별 저점 사유 상세

### 3.1 기술 구현 수준: 67/100

#### 감점 항목

| 세부 항목 | 감점 | 사유 |
|---|---:|---|
| 그래프 무결성 | -12 | warm-cache 분석 시 `db-table` 노드 유실 |
| API 실사용성 | -8 | `/api/graph/node/:nodeId`, `/impact`가 file-backed node에서 실패 |
| 분석 범위 완결성 | -7 | `route-renders`, event virtual edge, DTO flow, interface parsing 미완성 |
| 확장성/성능 구조 | -4 | `ParallelParser`가 실질 worker 기반이 아님, UI 번들 큼 |
| 설정 충실도 | -2 | `services[]`, `include[]`가 analyze/serve에 반영되지 않음 |
| **합계** | **-33** |  |

#### 상세 사유

1. **정확도보다 시연 가능성에 치우친 구현이 남아 있다**
   - cold run에서는 보이는 정보가 warm run에서 사라지는 것은 분석 도구에서 치명적이다.
   - 사용자는 캐시 사용 여부에 따라 다른 결과가 나오면 결과를 신뢰하지 못한다.

2. **일부 API가 문서상 기능과 실제 동작이 다르다**
   - slash가 포함된 node ID를 path param으로 운반하는 설계는 실운영 API에 부적합하다.
   - 상세 조회와 영향 분석 API는 분석 도구의 핵심 탐색 기능인데, 현재는 일부 노드에서만 작동한다.

3. **분석 도메인 완결성이 문서 수준까지 도달하지 못했다**
   - Vue router → component 렌더링 관계
   - 프론트 이벤트 emission/listener 관계
   - DTO 흐름 추적
   - MyBatis mapper interface parsing
   - 위 항목은 내부 영향 분석 정확도에 직접 영향을 준다.

4. **대규모 조직형 프로젝트 대응 구조가 아직 약하다**
   - monorepo/MSA 인식은 문서와 초기화 단계에 있으나 실제 분석 입력 모델이 따라오지 못한다.
   - worker 기반 병렬화, incremental robustness, large graph UX가 아직 제한적이다.

#### 상세 사유

1. **테스트 수는 많지만, 중요한 실패를 막지 못한다**
   - 성능 테스트가 cache hit 0이어도 통과한다.
   - route 테스트가 정상 실패를 known limitation으로 통과시킨다.

2. **문서가 살아 있는 제품 문서 역할을 아직 하지 못한다**
   - 문서 일부는 roadmap와 implementation summary가 혼합돼 있다.
   - 개발자/영업/운영 관점에서 읽을 때 실제 구현 범위를 오해하게 만든다.

3. **제품 문서와 검증 문서의 분리가 부족하다**
   - “현재 구현”과 “향후 계획”이 한 문서 안에서 섞이는 지점이 있다.

- 문서에 적은 분석 범위가 실제 구현과 일치
- 대형 프로젝트 분석 구조가 검증됨
- monorepo/MSA/alias/include/exclude가 실제 동작

- 치명적 결함을 회귀 테스트가 반드시 차단
- 문서가 roadmap와 current state를 명확히 분리
- 제품 문서, 관리자 문서, 사용자 문서, 테스트 명세가 정합적
- 모든 공개 API와 핵심 UX에 수용 기준이 존재

### 4.3 제품화 준비도 100점

- 인증, 권한, 감사, 관리, 설치, 업그레이드, 운영, 복구가 준비됨
- 폐쇄망/사내망 설치 패키지와 운영 매뉴얼이 존재
- 관리자 기능과 정책 제어 기능이 제공됨

### 4.4 금융권 내부 임직원용 도구 적합도 100점

- 조직/프로젝트/권한 중심의 운영 모델 지원
- 저장소 등록/승인/분석 이력/결과 공유가 가능
- 민감 정보 통제와 감사 대응이 가능
- 개발/운영/아키텍처 조직 모두가 활용 가능한 흐름 제공

- 위 4개 항목이 사실상 충족
- 보안 점검, 운영 점검, 사용자 수용시험, 성능 기준, 납품 산출물이 패키지화
- 납품 후 운영 책임 분담과 유지보수 모델이 정의됨

## 5. 100점 달성을 위한 요구사항 명세

## 5.1 공통 원칙

  - phase2~phase5 summary와 user-stories가 재감사 내용과 충돌하지 않는다.

### R-GOV-002. 모든 핵심 기능은 요구사항 ID와 테스트 ID를 가져야 한다

- 요구사항:
  - 기능 요구사항마다 고유 ID를 부여한다.
  - 대응 테스트 케이스와 문서 링크를 매핑한다.

## 5.2 기술 구현 수준 100점 요구사항

### A. 그래프 무결성

### R-TECH-001. 캐시 사용 여부와 무관하게 그래프 결과는 동일해야 한다

- 우선순위: P0
- 요구사항:
  - cold run, warm run, incremental update run의 node/edge set이 논리적으로 동일해야 한다.
  - synthetic node(`db-table`, virtual event node 등)도 cache entry에 완전 보존되어야 한다.
- 수용 기준:
  - 동일 프로젝트에 대해 cold/warm 결과 diff가 0
  - `db-table` 노드 warm-cache 유실 재현 불가

### R-TECH-002. node identity scheme을 URL-safe하고 stable하게 재설계해야 한다

- 우선순위: P0
- 요구사항:
  - path param에 절대경로 원문을 직접 싣지 않는다.
  - node ID는 내부 key와 외부 transport key를 분리한다.
  - 상세 조회/영향 분석 API는 query param 또는 opaque ID 기반으로 변경한다.
- 수용 기준:
  - 모든 node kind에 대해 detail/impact API 200 응답
  - slash, colon, unicode 포함 ID에 대한 contract test 통과

### R-TECH-003. node stats와 edge stats는 명확히 분리해야 한다

- 우선순위: P0
- 요구사항:
  - `getStats()` 또는 대체 API에서 `nodeKinds`, `edgeKinds`를 별도 구조로 반환한다.
  - CLI 출력도 이 구조를 반영한다.
- 수용 기준:
  - CLI `Node Types`에 edge kind가 출력되지 않음

### R-TECH-004. `services[]`, `include[]`, `exclude[]`가 실제 분석 경로에 반영되어야 한다

- 우선순위: P0
- 요구사항:
  - CLI와 server의 discover/watch 로직이 동일 config contract를 사용한다.
  - monorepo/MSA service root를 복수 개 인식한다.
  - include/exclude 패턴이 제품 전 경로에서 일관되게 적용된다.
- 수용 기준:
  - 복수 서비스 fixture에서 service 단위 분석 성공
  - include/exclude contract test 통과

### B. 분석 범위 완결성

### R-TECH-005. Vue router와 component rendering 관계를 실제 edge로 생성해야 한다

- 우선순위: P1
- 요구사항:
  - router definition에서 route → component edge를 생성한다.
  - lazy import route도 해석한다.
- 수용 기준:
  - `route-renders` edge 생성
  - unused component 분석이 실제 route usage를 반영

### R-TECH-006. frontend event virtual edge를 지원해야 한다

- 우선순위: P1
- 요구사항:
  - `defineEmits`, template listener, `emit()` 호출을 통합해 event graph를 만든다.
  - parent-child listener linkage를 추적한다.
- 수용 기준:
  - `emits-event`, `listens-event`가 fixture 기준으로 연결됨

### R-TECH-007. Java interface parsing과 MyBatis mapper interface linking을 완성해야 한다

- 우선순위: P1
- 요구사항:
  - `interface` 선언도 parser가 해석한다.
  - `@Mapper` interface와 XML namespace를 안정적으로 매칭한다.
- 수용 기준:
  - class/interface 혼합 fixture에서 MyBatis linkage 정확도 100%

### R-TECH-008. DTO flow 추적을 지원해야 한다

- 우선순위: P1
- 요구사항:
  - controller ↔ service ↔ mapper ↔ frontend API payload 간 DTO 흐름을 모델링한다.
  - 최소한 DTO class/TS type 단위 흐름 노드 또는 edge를 제공한다.
- 수용 기준:
  - 대표 API 호출에서 DTO flow 시각화 가능

## 7. 최종 판단

현재 점수가 낮은 이유는 “아이디어가 약해서”가 아니라, **시제품과 납품 제품 사이의 간극이 아직 크기 때문**이다.

이 프로젝트를 100점 수준으로 끌어올리려면 다음 4가지를 동시에 달성해야 한다.

1. 분석 결과의 무결성을 cold/warm/incremental 전 구간에서 보장할 것
2. 테스트와 문서가 실제 기능을 거짓 없이 반영할 것
3. 인증·권한·감사·운영·배포를 갖춘 제품으로 바꿀 것
4. 금융권 내부 조직이 실제로 사용할 수 있는 승인·통제·보고 모델을 제공할 것

즉, 남은 과제는 단순 기능 추가보다 **신뢰성, 운영성, 통제성, 증적성의 확보** 에 가깝다.
