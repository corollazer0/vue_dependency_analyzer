## 7. 현재 수준 평가

### 7.1 왜 67/100인가

높게 본 점:

- 엔진/CLI/서버/UI가 분리된 구조를 이미 보유
- 테스트 자산이 적지 않음
- 문서와 감사 이력이 누적되어 있음
- 실제로 돌아가는 시연이 가능함

낮게 본 점:

- warm-cache 무결성 결함 존재
- 일부 API의 실제 사용성 결함 존재
- 운영 통제 기능이 약함
- 인증/권한/감사/패키징이 부재
- 금융권 제품화에 필요한 보안·운영 산출물이 아직 부족

### 7.2 현재 단계 판정

현재 단계는 아래 중 **"내부 PoC~파일럿"** 으로 보는 것이 적절하다.

| 단계 | 판정 |
|---|---|
| 아이디어/문서 단계 | 아님 |
| 기술 검증 PoC | 해당 |
| 내부 파일럿 | 일부 해당 |
| 일반 상용 패키지 | 아직 아님 |
| 금융권 납품 패키지 | 아직 아님 |

## 8. 금융권 내부 의존성 분석 도구로서의 납품 가능성

### 8.1 판정

- **지금 당장 납품 가능: 아니오**
- **추가 보완 후 제한적 납품 가능: 예**

### 8.2 이유

이 제품은 범주 자체는 맞다.

- 코드 구조를 시각화하고
- 변경 영향 범위를 파악하며
- 프론트/백/DB 의존성을 추적해
- 개발자·운영자·아키텍트의 분석 시간을 줄일 수 있다.

즉, 금융권 내부 임직원의 개발·운영을 돕는 **개발지원 툴**이라는 관점에서는 방향이 맞다.

하지만 현재 상태 그대로는 아래 이유로 납품하기 어렵다.

1. 내부통제 기능이 부족하다.
   - 사용자 인증, 권한관리, 접속이력, 감사로그가 없다.
2. 운영 제품으로서의 완결성이 부족하다.
   - 설치 패키지, 관리자 기능, 백업/복구, 장애대응 가이드가 약하다.
3. 현재 감사에서 기능 무결성 이슈가 남아 있다.
   - warm-cache DB layer 유실
   - 일부 node detail API 사용 불가
4. 금융권 보안 심사 대응 자산이 부족하다.
   - 취약점 관리, 오픈소스 관리, SBOM, 폐쇄망 배포 기준, 운영 보안 기준 대응이 미흡하다.

## 11. 최종 의견

이 프로젝트는 "쓸모 있는 시제품"이다.  
하지만 "금융권 내부 임직원용으로 바로 납품 가능한 솔루션"은 아직 아니다.

경영 관점에서 보면 가장 현실적인 판단은 다음과 같다.

- 현재 자산의 기술 가치는 **분명히 있다**
- 공수 환산으로도 **12.5 MM** 수준의 실체가 있는 저장소다
- 다만 **상품화와 금융권 납품 수준** 으로 끌어올리려면, 추가로 **8~12 MM** 정도의 안정화·보안·운영화 투자가 필요하다

따라서 이 프로젝트는 폐기 대상이 아니라, **금융권 내부 개발지원 도구로 제품화할 가치가 있는 1차 자산** 으로 보는 것이 적절하다.
